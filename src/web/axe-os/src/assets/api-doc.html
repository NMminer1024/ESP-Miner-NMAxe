AxeOS API Reference

This document describes the HTTP and WebSocket API exposed by the ESP32-based NMAxe / NMAxeGamma / NMQAxe++ firmware. All endpoints are served on port 80. JSON is used for all request and response bodies unless noted otherwise.

🌐 Base URL  http://<device-ip>
📦 Content-Type  application/json
🔓 Auth  None
🔄 CORS  * (all origins allowed)
⚙️
System

Core device management endpoints: read live status, restart the miner, and clear block-hit counters.

GET /api/system/info Live device status snapshot

Returns a comprehensive status snapshot: power measurements, temperatures, ASIC configuration, mining statistics, board identity, active stratum pool, and fan speeds. Polled every few seconds by the AxeOS dashboard.

Response 200 – application/json
{
  "power": {
    "power":  12.5,          // Watts (vbus_V × ibus_A)
    "vbus":   5020,          // mV – input bus voltage
    "ibus":   2490           // mA – input bus current
  },
  "temps": {
    "vcore": 45.2,           // °C – voltage-regulator temperature
    "asic":  52.8            // °C – ASIC die temperature
  },
  "asic": {
    "count":        1,      // number of ASIC chips detected
    "model":        "BM1366",
    "vcoreReq":     1200,   // mV – requested vcore
    "vcoreReal":    1198,   // mV – measured vcore
    "freqReq":      485,    // MHz – requested frequency
    "smallCoreCnt": 894    // number of small cores
  },
  "miner": {
    "hashRate":        0.465,      // GH/s – 3-minute rolling average
    "bestDiffEver":    "52.3G",    // best difficulty since first boot
    "bestDiffSession": "12.1G",    // best difficulty this session
    "networkDiff":     "83.67T",   // current Bitcoin network difficulty
    "poolDiff":        "8192",     // pool-assigned share difficulty
    "lastDiff":        "8192",     // difficulty of the last accepted share
    "blkhits":         0,          // block hits (shares ≥ network diff)
    "freeHeap":        204800,    // bytes of free heap
    "sAccepted":       1024,      // accepted shares this session
    "sRejected":       3,          // rejected shares this session
    "uptimeSeconds":   3600,      // seconds since last reboot
    "uptimeEver":      86400      // cumulative uptime seconds
  },
  "identity": {
    "fwVersion": "1.2.0",
    "hwModel":   "NMAxe",
    "hostName":  "nmaxe-a1b2",
    "ssid":      "MyHomeWiFi",
    "rssi":      -62           // dBm – WiFi signal strength
  },
  "stratum": {
    "url":  "stratum+tcp://public-pool.io:21496",
    "user": "bc1q...YOUR_ADDRESS...xyz.nmaxe",
    "pwd":  "x"
  },
  "fans": [
    { "id": 0, "speed": 80, "rpm": 4200 },  // ASIC fan
    { "id": 1, "speed": 60, "rpm": 3100 }   // vcore fan

POST /api/system/restart Restart the device

Triggers a software restart (ESP.restart()). No request body is required.

⚠️ The device will become unreachable for ~10–15 seconds while it reboots.

POST /api/system/clearhits Reset the block-hit counter

Resets the blkhits counter (shares whose difficulty meets or exceeds the current network difficulty) to zero. Mining continues uninterrupted.

Request body
// empty body or {}
Response 200

GET /api/wakeup Wake up the screensaver

Resets the screensaver idle timer and clears the SYS_EVENT_SCREEN_SAVER_TRIGGERED event bit, waking the device display. Called by the dashboard and swarm pages periodically to keep the screensaver inactive while the user is monitoring. Setting and update pages do not call this endpoint, allowing the screensaver to activate during long idle sessions.

Response 200
{ "ok": true }

GET /api/dashboard/chart/history Full miner status time-series

Returns historical miner statistics as a time-series array (up to 2 000 data points). Used to populate the history charts on the Monitor page.

Query paramTypeDescription

POST /api/update/screensaver Upload custom screensaver GIF

Upload a custom animated GIF to replace the built-in screensaver. The file is saved to SPIFFS as screen_saver_240x135.gif (NMAxe / NMAxeGamma) or screen_saver_320x240.gif (NMQAxe++). The new GIF is used on the next screensaver trigger. Only .gif files are accepted; any other extension is rejected with 400.

ℹ️ Upload progress is tracked by the firmware in g_board.status.ota and shared by all upload types (firmware, SPIFFS, and screensaver GIF). Poll GET /api/update/progress to get accurate device-side write progress during the upload.
Request – multipart/form-data
POST /api/update/screensaver HTTP/1.1
Content-Type: multipart/form-data; boundary=----boundary

------boundary
Content-Disposition: form-data; name="screensaver"; filename="my-animation.gif"
Content-Type: application/octet-stream

<binary GIF data>
------boundary--
Response 200
{ "status": "ok" }
Response 400 – non-GIF file
Only .gif files are accepted.
Response 500 – SPIFFS write failure
Write error.

OTA Update

Over-the-air firmware and SPIFFS (web UI) flash endpoints. Both canonical URLs and legacy compatibility aliases are supported.

⚠️ The device will reboot automatically after a successful flash. Ensure the uploaded binary matches the board model. Partial or interrupted uploads leave the device in recovery mode — re-uploading recovers it.
GET /api/update/progress Poll upload / flash progress

Returns the current state of the unified upload handler. Applies to all upload types: firmware (firmware.bin), SPIFFS image (spiffs.bin), and screensaver (.gif). The frontend polls this endpoint (e.g. every 500 ms) during any upload to display accurate device-side write progress. Progress is tracked in g_board.status.ota and updated by file_upload_handler.

Response 200 – application/json
{
  "running":  true,           // true while an upload/flash is in progress
  "progress": 45,             // 0–100 (%)
  "filename": "firmware.bin"  // name of the file currently being written
}
ℹ️ Poll this endpoint during any upload to track progress. When running is false the previous upload has completed (or no upload has started). The device reboots automatically after a successful firmware or SPIFFS flash. Screensaver GIF uploads do not require a reboot.
POST /api/update/firmware Flash new firmware (.bin)

Upload a new firmware binary via multipart/form-data. The device will flash it to the OTA partition and reboot. Legacy alias: /api/system/OTA

Request – multipart/form-data
POST /api/update/firmware HTTP/1.1
Content-Type: multipart/form-data; boundary=----boundary

------boundary
Content-Disposition: form-data; name="file"; filename="firmware.bin"
Content-Type: application/octet-stream

<binary firmware data>
------boundary--
Response 200 (on success, before reboot)
{ "status": "ok" }
Response 400 (flash failed)
{ "error": "flash failed" }
POST /api/update/spiffs Flash new web UI (SPIFFS image)

Upload a new SPIFFS filesystem image. The device sets an NVS flag before writing and clears it on success; an interrupted upload leaves the flag set, forcing recovery mode on the next boot. Legacy alias: /api/system/OTAWWW

Request – multipart/form-data
POST /api/update/spiffs HTTP/1.1
Content-Type: multipart/form-data; boundary=----boundary

------boundary
Content-Disposition: form-data; name="file"; filename="spiffs.bin"
Content-Type: application/octet-stream

<binary SPIFFS image>
------boundary--
Response 200
{ "status": "ok" }

Swarm

Swarm mining aggregates statistics across multiple NMAxe devices on the same LAN. The background alive_ip_scan_thread periodically pings the /24 subnet; swarm_thread probes found IPs for NM-compatible firmware.

POST /api/swarm/scan Trigger an immediate subnet re-scan

Signals the IP-scan thread to abort the current scan and restart from the beginning. Typically called by the AxeOS Swarm page on load to get fresh data quickly. No request body is needed.

Logging

Real-time log output is pushed over the persistent WebSocket connection at /ws. The HTTP GET /api/log endpoint is a lightweight probe that confirms the log channel is available; actual log lines are delivered as WebSocket text frames.

GET /api/log Log feature probe / trigger

A lightweight probe endpoint. Invoking it triggers an internal log event so the client can confirm the log channel is alive. Actual log lines are delivered as WebSocket text frames over ws://<device>/ws — not in this HTTP response.

Response 200
// 200 OK with empty body — real logs arrive via WebSocket /ws
WS ws://<device>/ws Real-time log & event stream

A persistent WebSocket connection served on port 80, path /ws. The firmware pushes every log line as a UTF-8 text frame in real time. Text sent by a client is echoed back verbatim (useful for connectivity testing). webSocket.cleanupClients() runs every 250 ms inside the webserver thread to detect and evict stale connections.

ℹ️ Connect with any standard WebSocket client: the browser WebSocket API, wscat, or the AxeOS built-in Log page.
Connection URL
ws://<device-ip>/ws

Probe & Discovery

Lightweight discovery endpoints used by the Swarm panel to identify NM-compatible miners and enumerate live hosts on the subnet.

GET /probe NM miner identification probe

Designed to be called by other devices (e.g. the Swarm panel of a neighbouring miner) to identify this device as an NM-compatible miner and retrieve its real-time statistics. A device is classified as NM-compatible if the response contains both hr and ver keys.

Response 200
{
  "model": "NMAxe",            // board model name
  "hostname": "nmaxe-a1b2",    // mDNS hostname
  "ver": "1.2.0",             // firmware version — mandatory for NM identification
  "sw": 135,                // TFT screen width in pixels
  "sh": 240,                // TFT screen height in pixels
  "hr": 465000000,            // H/s – 3-minute average — mandatory for NM identification
  "sbd": 12884901888,        // best session difficulty (raw)
  "ebd": 52428800000,        // best ever difficulty (raw)
  "ut": 3600                 // uptime in seconds for current session
}
GET /alive List of alive IPs on the subnet

Returns the device's own IP along with all IP addresses that responded to ICMP pings during the last /24 subnet scan. Rather than having each miner independently probe the network, the design offloads discovery to the querying side: the miner performs a single coarse ICMP sweep and caches the results; callers receive the full alive-IP list and are responsible for the finer-grained NM-device identification (via /probe). This keeps the miner's CPU and network load minimal — it only does one pass of raw reachability, leaving protocol-level classification to the client.

Response 200
{
  "self": "192.168.1.42",
  "ips":  [
    "192.168.1.42",   // self is always first
    "192.168.1.1",
    "192.168.1.100",
    "192.168.1.101"
  ]
}
